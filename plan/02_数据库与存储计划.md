## Harness（先做）

### H1：DB Migration Harness（P0）

- 目标：迁移可重复执行；核心表与索引存在
- 测试用例：
  - `should_create_all_core_tables` - 迁移后所有核心表（Tenant/User/TestTask/AgentInstance/TestStep/HilTicket）存在
  - `should_create_tenant_isolation_indexes` - 租户隔离索引创建成功
  - `should_enforce_tenant_id_on_queries` - 查询必须包含 tenant_id 条件
  - `should_rollback_migration_on_failure` - 迁移失败时能正确回滚

### H2：Storage Access Harness（P0）

- 目标：对象存储访问必须受控（预签名/代理）
- 测试用例：
  - `should_require_auth_for_object_access` - 无鉴权无法下载对象
  - `should_reject_expired_presigned_url` - 预签名过期不可访问
  - `should_reject_cross_tenant_object_access` - 跨租户 key 不能访问
  - `should_cleanup_expired_objects` - 过期对象能被清理

### H3：Data Lifecycle Harness（P1）

- 目标：保留期到期后资产不可访问且被清理

## TDD 步骤

1) 先写迁移后表结构的断言测试
2) 写 DAO/Repository 层测试（以 tenant_id 为强制条件）
3) 写对象存储接口的单测（使用 mock server 或 fake client）
4) 再实现实际连接与配置

## 设计原则（SOLID）

- Repository 层只负责数据访问，不包含业务决策（SRP）
- StorageClient 通过接口注入，便于 mock（DIP/ISP）

## 交付物

- DB 迁移脚本
- Repository/DAO
- StorageClient（S3/MinIO）
- Harness：迁移/访问/生命周期测试

## 验收标准

- tenant 隔离在 DB 查询层不可绕过
- 对象存储 URL 不可裸奔；访问必须鉴权
