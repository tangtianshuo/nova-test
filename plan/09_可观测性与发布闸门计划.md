# Plan 09：可观测性与发布闸门计划

## 目标

- 提供实例级可观测性：trace_id/instance_id 贯穿日志、事件、报告
- 建立发布闸门：契约/回放/安全/性能 基线

## 依赖关系

> **此计划是最终验收计划，依赖所有前置计划完成**

| 依赖计划 | 依赖原因 |
|---------|---------|
| Plan 01（项目基础设施） | 需要测试框架、CI/CD 配置 |
| Plan 02（Schema 定义） | Schema Harness 校验依赖 Schema 定义 |
| Plan 03（Step Runtime） | Traceability Harness 需追踪 Step 执行 |
| Plan 04（StateMachine） | StateMachine Node Harness 依赖节点定义 |
| Plan 05（Action System） | Regression Harness 需回放 Action 执行 |
| Plan 06（HIL Flow） | HIL Flow Harness 依赖决策流程 |
| Plan 07（Sensitive Data） | Sensitive Leak Harness 依赖脱敏机制 |
| Plan 08（UI Rendering） | UI No-Scroll Harness 依赖 UI 渲染完成 |

## 范围

### 可观测性（P0/P1）

- P0：结构化日志（带 tenant_id/instance_id/trace_id）
- P1：指标（步数、耗时、失败率、HIL 次数、推流断开率）
- P1：告警（失败、重试过多、额度不足）

## Harness（先做）

### H1：Traceability Harness（P0）

- 任意 Step/事件/报告都能回溯到 instance_id
- 任意错误日志都带 trace_id

**测试用例：**

| 用例名 | 描述 | 验证点 |
|-------|------|-------|
| `should_trace_step_to_instance` | 验证 Step 执行记录可追溯到 instance_id | Step 日志包含 `instance_id`、`step_id`、`timestamp` |
| `should_trace_event_to_instance` | 验证事件可追溯到 instance_id | 事件包含 `instance_id`、`event_type`、`trace_id` |
| `should_trace_report_to_instance` | 验证报告可追溯到 instance_id | 报告包含 `instance_id`、`report_id`、生成时间 |
| `should_include_trace_id_in_error_logs` | 验证错误日志包含 trace_id | 错误日志必须包含 `trace_id`、`error_code`、`stack_trace` |
| `should_include_tenant_id_in_all_logs` | 验证所有日志包含 tenant_id | 所有日志条目必须包含 `tenant_id` 字段 |

### H2：Regression Harness（P1）

- 固定 fixtures 回放生成报告，与 golden 对比
- 推流事件序列回放验证 UI/导出一致

**测试用例：**

| 用例名 | 描述 | 验证点 |
|-------|------|-------|
| `should_match_golden_report_on_replay` | 回放固定 fixtures 生成报告与 golden 一致 | 报告内容与 golden file diff 为零（忽略时间戳字段） |
| `should_detect_schema_breaking_change` | 检测 Schema 破坏性变更 | Schema 版本变更时自动检测 backward compatibility |
| `should_verify_ui_state_on_replay` | 回放事件序列验证 UI 状态一致性 | UI 截图与 golden 截图相似度 ≥ 99% |

### H3：Performance Harness（P1）

- 单步延迟基线（2~5s 目标的模拟/统计）
- 并发实例基线（至少在 mock 环境测出瓶颈）

**测试用例：**

| 用例名 | 描述 | 验证点 |
|-------|------|-------|
| `should_complete_step_within_time_limit` | 验证单步执行在时间限制内完成 | 单步延迟 ≤ 5s（P95），≤ 2s（P50） |
| `should_handle_concurrent_instances` | 验证并发实例处理能力 | 10 个并发实例无资源竞争、无死锁、内存稳定 |
| `should_measure_streaming_latency` | 验证推流延迟指标 | 推流首帧延迟 ≤ 500ms，断开后重连 ≤ 2s |

## 发布闸门（DoD）

| Harness | 通过条件 | 具体验证点 |
|---------|---------|-----------|
| Schema Harness | 全绿 | 7 个 Schema 校验通过：`StepSchema`、`EventSchema`、`ReportSchema`、`ActionSchema`、`StateSchema`、`ConfigSchema`、`UISchema` |
| API Contract Harness | 全绿 | 所有错误码覆盖：`400/401/403/404/429/500/503`，每个错误码有对应的 response schema |
| StateMachine Node Harness | 全绿 | 5 个节点通过：`StartNode`、`ActionNode`、`DecisionNode`、`HILNode`、`EndNode` 的状态转换校验 |
| Replay Harness | 全绿 | 4 种回放场景通过：`happy_path`、`error_recovery`、`hil_approval`、`streaming_events` |
| HIL Flow Harness | 全绿 | 3 种决策通过：`approve`、`reject`、`timeout_fallback` |
| Sensitive Leak Harness | 全绿 | 4 种模式扫描通过：`log_leak`、`report_leak`、`screenshot_leak`、`export_leak` |
| UI No-Scroll Harness | 全绿 | 4 个分辨率通过：`1920x1080`、`1366x768`、`1280x720`、`mobile_viewport` |
| Golden Files | 变更必须审查 | 变更需记录：变更原因、影响范围、审批人、更新时间 |

## 交付物

- 日志规范与实现
- 指标/告警（如有）
- CI 中的闸门配置（执行上述 harness）

## 验收标准

- 能用 instance_id 一键定位：截图、动作、验证、HIL 决策、报告
- 任何破坏契约/回放/安全的变更无法通过闸门
