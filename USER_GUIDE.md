## 系统概述

Nova Test 是一个智能化的黑盒自动化测试平台，通过 AI Agent 自动执行 Web 应用测试：

- **自然语言描述测试目标**：使用日常语言描述测试需求，无需编写代码
- **AI 自动生成测试步骤**：Vision 模型分析页面并规划动作序列
- **实时可视化监控**：WebSocket 推送执行状态、截图和日志
- **人工审核介入**：关键决策点可暂停执行进行人工确认

### 典型使用场景

1. **回归测试**：自动化验证新版本功能是否正常
2. **冒烟测试**：快速验证核心业务流程
3. **探索性测试**：AI 自主探索页面发现潜在问题
4. **持续集成**：与 CI/CD 流水线集成自动化测试

## 任务管理

### 创建任务

#### 步骤 1: 基本信息

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| 任务名称 | ✅ | 便于识别的任务标识 | "电商购买流程测试" |
| 目标 URL | ✅ | 被测网站的起始页面 | `https://shop.example.com` |
| 测试目标 | ✅ | 自然语言描述测试目标 | "搜索 MacBook，加入购物车并结算" |

#### 步骤 2: 配置约束（可选）

| 约束项 | 说明 | 示例 |
|--------|------|------|
| 最大步数 | 限制测试执行的最大动作数 | `20` |
| 禁止域名 | 测试过程中禁止访问的域名 | `["payment.com", "stripe.com"]` |
| 超时时间 | 单步执行超时限制（秒） | `30` |
| 重试次数 | 动作失败时的重试次数 | `2` |

#### 步骤 3: 高级选项

- **前置条件**：指定进入目标页面的前置操作
- **期望结果**：指定测试成功/失败的判断标准
- **标签**：为任务添加分类标签便于管理

#### 创建示例

```json
{
  "name": "电商搜索购买测试",
  "target_url": "https://shop.example.com",
  "natural_objective": "搜索关键词 'iPhone'，点击第一个商品进入详情页，添加到购物车",
  "constraints": {
    "max_steps": 15,
    "forbidden_domains": ["payment.com", "alipay.com", "wechat.com"],
    "timeout_seconds": 45,
    "retry_count": 2
  }
}
```

### 管理任务

#### 查看任务列表

- 支持按状态筛选：`全部` / `运行中` / `已暂停` / `已归档`
- 支持按创建时间、名称排序
- 支持关键字搜索

#### 编辑任务

点击任务卡片右上角「编辑」按钮：

- 修改基本信息
- 调整约束配置
- 更新测试目标描述

#### 任务状态

| 状态 | 说明 | 可执行操作 |
|------|------|------------|
| `ACTIVE` | 任务活跃，可创建实例执行 | 创建实例、编辑、暂停 |
| `PAUSED` | 任务暂停 | 恢复、编辑、归档 |
| `ARCHIVED` | 已归档，仅供查看 | 恢复、删除 |

### 批量操作

- **批量启动**：选择多个任务同时创建实例
- **批量暂停**：选择多个任务同时暂停
- **批量归档**：选择多个任务同时归档

## HIL 人机协作

### 什么是 HIL

Human-in-the-Loop (HIL) 机制允许在 AI 执行过程中引入人工审核：

- **高风险操作识别**：AI 自动识别可能产生重大影响的动作
- **置信度检查**：低置信度决策自动暂停等待确认
- **人工干预**：用户可以批准、拒绝或修改 AI 建议的动作

### HIL 触发场景

| 触发条件 | 风险等级 | 说明 |
|----------|----------|------|
| 置信度 < 0.7 | MEDIUM | AI 对决策不够自信 |
| 置信度 < 0.5 | HIGH | AI 对决策非常不确定 |
| 支付/登录操作 | HIGH | 涉及敏感操作 |
| 禁止域名访问 | HIGH | 违反约束规则 |
| 解析失败 | MEDIUM | 无法理解页面内容 |

### 审核操作

#### 批准执行

点击「批准」直接执行 AI 建议的动作。

#### 拒绝

点击「拒绝」跳过当前步骤，继续下一步骤。

#### 修改动作

点击「修改」打开参数编辑：

| 参数 | 说明 |
|------|------|
| 动作类型 | click / type / navigate / screenshot |
| 选择器/目标 | CSS 选择器或 URL |
| 输入值 | type 动作的输入内容 |

### 超时设置

可在任务约束中配置 HIL 超时时间：

- **超时时间**：默认 5 分钟
- **超时操作**：可选择「自动批准」或「自动拒绝」
- **通知设置**：超时前发送提醒通知

## 最佳实践

### 测试目标描述

#### ✅ 推荐写法

- "在搜索框输入 'iPhone'，点击搜索按钮"
- "登录后进入用户中心，点击修改密码"
- "浏览商品列表，点击第一个商品进入详情"

#### ❌ 避免写法

- "测试登录功能"（太模糊）
- "执行所有操作"（不明确）
- "检查页面是否正常"（无法判断成功/失败）

### 约束配置

#### 安全约束

```json
{
  "forbidden_domains": [
    "payment.com",
    "alipay.com",
    "wechat.com",
    "bank.com"
  ]
}
```

> 💡 **建议**：支付相关域名务必加入禁止列表，避免产生实际交易

#### 步数控制

```json
{
  "max_steps": 20
}
```

> 💡 **建议**：根据测试目标复杂度设置合理步数，一般简单流程 10-15 步，复杂流程 30-50 步

### HIL 配置

| 场景 | 推荐阈值 | 说明 |
|------|----------|------|
| 高风险场景 | 0.9 | 严格审核所有决策 |
| 标准测试 | 0.7 | 平衡效率和安全性 |
| 快速验证 | 0.5 | 仅高不确定性时介入 |

### 常见问题处理

#### 页面加载超时

- 增加 `timeout_seconds` 约束
- 在测试目标中描述等待条件

#### 元素定位失败

- AI 会自动尝试多种定位策略
- 可在 HIL 中手动指定正确的选择器

#### 误报缺陷

- 检查 verify 节点的验证逻辑
- 可在测试目标中明确期望结果
