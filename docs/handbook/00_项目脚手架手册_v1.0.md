# SaaS + AaaS + SDD + Harness Engineering 项目脚手架手册 (v1.0)

本手册用于把本仓库中讨论/生成的文档、规则与计划“打包成一套可复制的项目初建脚手架”，可直接迁移到其他项目作为起步标准。

## 1. 适用场景

- 你要做一个 **SaaS 平台**（多租户、RBAC、审计、计费/额度）
- 平台提供 **AaaS（Agent as a Service）**：实例化 Agent 执行任务
- Agent 具备 **视觉感知（VLM/CUA）**：截图 -> 决策 -> 执行 -> 验证
- 系统需要 **Human-in-the-Loop（HIL）**：高危/低置信度/异常时人工介入
- 研发范式为 **SDD（Schema Driven Development）** + **Harness Engineering** + **TDD**
- UI 要求：**全屏无滚动条**（分页/折叠/抽屉替代滚动）

## 2. 你应该复制哪些内容（最小脚手架包）

将以下目录/文件复制到新项目（建议保持路径不变）：

```text
docs/
  design/                # 设计体系与规范
  handbook/              # 本手册
  demo/                  # 单页静态Demo（全屏无滚动条）
plan/                    # 多Plan执行计划（Harness + TDD）
.trae/skills/            # Rule Skill 与 Agent Prompts
```

复制后，你需要做的最小改动：
- 全局替换项目名（例如“黑盒自动化测试工具”）
- 调整 `docs/design/00_文档索引与规范_v1.0.md` 的文档清单
- 调整 `plan/00_INDEX.md` 的计划列表与全局约束（保持 Harness/TDD/SDD 不变）

## 3. 文档体系（可复制模板）

### 3.1 设计文档（建议作为新项目的“初始必备”）

推荐保留并在新项目中改写内容：
- 技术可行性/技术路径（Feasibility/Tech Path）
- 需求设计、概要设计、功能设计、数据库设计、详细设计、接口设计
- UI 视觉规范、UE 交互规范、颜色设计
- 详细需求文档（带验收标准）
- 开发任务文档与任务表（可导入项目管理工具）
- 视觉感知与可视化说明（说明 Observe-Think-Act 与标注层展示）
- Rule Skill 与 Agent 提示词（AaaS 与开发阶段 Harness）

### 3.2 索引规范

- 统一放置：`docs/design/`
- 命名：`NN_文档名称_vX.Y.md`
- 版本：
  - `v1.0` 首版
  - `v1.1` 兼容性增补
  - `v2.0` 破坏性变更

## 4. 开发范式（必须遵循）

### 4.1 SDD（Schema 先行）

跨模块对象必须先定义 Schema，再写实现。

最低要求：
- 所有对象带 `schema_version`
- 所有输出必须可解析为 JSON（或结构化文本），不允许“自由文本协议”

核心对象（建议固定命名）：
- `Task` / `Instance` / `Step`
- `Action`（Vision 输出 / Executor 输入）
- `Verify`（Verifier 输出）
- `Event`（推流事件）
- `HilTicket`
- `Report`

### 4.2 Harness Engineering（护栏优先）

把系统关键风险变成可自动验证的护栏（Harness）：
- **Schema Harness**：所有输入/输出可校验
- **API Contract Harness**：核心 API 契约测试（含错误码）
- **StateMachine Harness**：节点级单测 + 回放
- **HIL Harness**：触发/处理/恢复执行可测
- **Multi-tenant Harness**：跨租户负向测试
- **Storage Harness**：对象存储鉴权与生命周期
- **UI No-Scroll Harness**：全屏无滚动条断言 + 分页边界

### 4.3 TDD（测试驱动开发）

关键路径一律：
1) 先写失败测试（契约/回放/安全/边界）
2) 写最小实现让测试通过
3) 重构到 SOLID（不改测试）

## 5. 多租户落地建议（可复制的默认策略）

推荐默认：**单库多租户（tenant_id + 强制隔离）**，按规模升级：
- 阶段A（默认）：共享 DB + tenant 逻辑隔离（必要时 RLS）+ 热表按时间分区
- 阶段B（大租户）：大租户独立库/独立schema（控制面共享，数据路由）
- 阶段C（强合规/私有化）：专属部署（独立容器 + 独立数据库 + 专属对象存储）

选择依据（简单规则）：
- 合规要求需要物理隔离 -> 阶段C
- 单租户压垮共享资源/清理困难 -> 阶段B
- 其他情况优先阶段A（成本最低、迭代最快）

## 6. UI 脚手架规范（全屏无滚动条）

硬性规则：
- `html, body` 必须 `height: 100%` 且 `overflow: hidden`
- 禁止任何面板滚动条（不使用 `overflow:auto/scroll`）
- 超量信息用：分页/折叠/抽屉/切换视图
- JSON/长文本：截断预览 + 一键复制全文

Demo 参考：
- `docs/demo/index.html`（已按全屏无滚动条实现：列表/日志/时间线均分页）

## 7. Agent 体系脚手架（AaaS）

### 7.1 运行闭环

Observe（截图） -> Think（Vision 决策） -> Act（Executor 执行） -> Verify（Verifier 断言）

### 7.2 HIL 策略（默认）

必须触发 HIL：
- 低置信度
- 高危动作（支付/删除/发送等）
- JSON 解析失败/不符合 Schema
- 异常循环/超时

### 7.3 可视化

用户应能看到“视觉感知过程”：
- 推流截图 + 标注层（bbox/point/候选框）
- thought/confidence/expected_result
- HIL 工单（原因/风险/动作/截图标注）

## 8. Skill（可复制的工程化能力）

将 `.trae/skills/` 作为“团队规则与提示词资产库”，建议最少包含：
- `rule-sdd-aaas`：SaaS+AaaS+SDD 规则清单
- `agent-prompts-aaas`：AaaS 运行期多 Agent 提示词
- `rule-harness-dev`：开发阶段 Harness 规则与闸门
- `agent-prompts-harness-dev`：开发阶段多 Agent（编排/实现/测试/安全/评审）提示词

## 9. Plan（可复制的多阶段执行计划）

将 `plan/` 作为新项目的“实施路线图”，每个 Plan 都必须包含：
- 目标与范围
- Harness（先做）
- TDD 步骤
- SOLID/安全/多租户约束
- 交付物与验收标准

建议 Plan 列表（可直接复制）：
- SDD 与 Schema
- 数据库与存储
- 控制面 API 与鉴权
- AaaS 执行面与状态机
- 推流与 HIL
- 报告与导出
- UI 控制台（全屏无滚动）
- 安全合规与多租户验证
- 可观测性与发布闸门

## 10. 新项目 Day-0 初始化清单（复制即用）

- 复制 `docs/`、`plan/`、`.trae/skills/`
- 改项目名、改域名/URL 示例
- 明确技术栈与部署形态（共享 DB vs 专属部署）
- 从 `plan/01` 开始推进：先 Schema + Schema Harness

## 11. 代码质量与中文注释要求（落地规范）

### 11.1 SOLID + 迪米特（最低落地要求）

- 业务服务不直接依赖具体 SDK（依赖倒置：依赖接口）
- 一个类/模块只做一类事情（单一职责）
- 策略可插拔（开闭原则）
- 子类型可替换父类型（里氏替换）
- 接口尽量小（接口隔离）
- 减少跨层对象“穿透调用”（迪米特）

### 11.2 中文注释（关键点）

以下位置必须写中文注释说明“意图/风险/兜底策略”：
- 多租户隔离与权限校验
- HIL 触发条件与恢复执行
- 状态机路由与终止/重试策略
- 敏感信息处理与脱敏
