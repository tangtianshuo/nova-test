# 黑盒自动化测试工具 - 视觉感知与可视化设计说明 (v1.0)

## 1. 视觉感知能力的实现路径

本系统的“视觉感知”并非传统的 DOM/Accessibility 解析，而是采用 **VLM/CUA（Computer Use Agent）** 路线：
- **输入**：当前页面截图（Image）+ 历史动作与上下文（Text）+ 测试目标（Text）
- **输出**：下一步动作计划（Action JSON），包含动作类型、目标位置（坐标/候选框）、置信度、预期结果描述

核心组件：
- **Fara-7B**：作为视觉推理与决策模型（负责“看懂页面并决定做什么”）
- **Magentic-UI / Playwright**：作为执行器（负责“按模型给的动作去点/输/滚动”）
- **LangGraph**：作为状态机编排器（负责“循环执行、容错、阻断、人机协同”）

## 2. 视觉感知的运行时数据流（Observe-Think-Act）

### 2.1 Observe（观察：获取画面与上下文）
1. Magentic-UI 在沙箱浏览器中执行到某一步后，抓取当前 **截图**。
2. 截图做基础处理（建议）：
   - 缩放到固定长边（例如 1280）以平衡速度与精度
   - 压缩与编码（WebP/JPEG），同时保留原图用于报告
3. 构造上下文：
   - `objective`：自然语言测试目标
   - `history`：最近 N 步的动作摘要（避免 token 爆炸）
   - `constraints`：禁止域名、最大步数、敏感操作策略等

### 2.2 Think（思考：视觉推理与动作规划）
1. 将 `image + context` 发送给 Fara-7B 推理服务。
2. 推理端强制输出为 **SDD JSON**（ActionSchema），避免“自由文本不可解析”。
3. 模型输出建议结构（示例字段）：
   - `thought`：推理摘要（可展示给用户）
   - `action_type`：click/type/scroll/wait/end
   - `target`：
     - `x,y`（点击坐标）或
     - `bbox`（候选框：x1,y1,x2,y2）或
     - `selector`（可选 fallback）
   - `confidence`：0~1
   - `expected_result`：期望页面变化

### 2.3 Act（行动：浏览器执行与反馈闭环）
1. LangGraph 将 `planned_action` 交给 Magentic-UI 执行。
2. 执行后抓取新截图并进入 Verify（验证）。
3. 验证可采用两种策略（可组合）：
   - **视觉断言**：Fara-7B 对比 before/after 截图 + expected_result 给出 `isSuccess`/`isDefect`
   - **结构化辅助**（可选）：DOM/网络状态作为补充信号（提高稳定性）

## 3. 视觉感知的关键工程实现点

### 3.1 推理服务形态
- **独立推理服务**：Fara-7B 部署在 GPU 节点（vLLM/TGI/原生 Transformers），对外提供 HTTP/RPC。
- **多租户隔离**：推理服务不持久化租户截图；仅处理请求并返回结构化结果。
- **吞吐与并发**：通过队列与批处理提升吞吐；为不同租户/套餐设置并发配额。

### 3.2 输出稳定性（SDD 关键）
- 强制 JSON 输出模式（或服务端对输出做 JSON 修复/重试）。
- 置信度低或 JSON 解析失败时，进入 `HIL` 阻断：让人确认或修改动作。

### 3.3 隐私与敏感信息
- 截图属于高敏数据：
  - 存储加密、设置保留期、按租户隔离
  - UI 展示支持局部马赛克（如密码框区域）
- Prompt 上下文中避免注入明文凭证：账号/密码通过“变量注入”仅在执行器端使用。

## 4. 用户是否能直观看到“视觉感知过程”？

结论：**可以**，且推荐以“可解释 + 可回放”的方式呈现。

### 4.1 实时可视化（运行中）
在实例执行详情（Live Console）中让用户看到：
- **实时截图推流**：每步更新当前画面
- **目标标注层（Overlay）**：在截图上绘制模型认为的目标区域
  - 点击：显示十字/圆点 + 可选小框
  - 候选框：显示 bbox 边框
  - 多候选：按排名绘制 1~3 个候选框
- **置信度与风险提示**：低置信度用 warning 色标记，并给出“建议人工确认”按钮
- **Thought/Expected Result**：展示模型推理摘要与预期变化（增强可解释性）

对应推流事件建议：
- `SCREENSHOT_UPDATE`：包含压缩后的 `image_b64` 或可访问 URL
- `AGENT_THOUGHT`：thought/confidence/expected_result
- `ACTION_PROPOSAL`（建议新增）：包含 bbox/坐标/候选列表（用于绘制标注层）

### 4.2 回放可视化（报告中）
在报告与 Timeline 中让用户看到：
- 每一步的“截图 + 标注 + 动作 + 结果”
- 失败步骤的对比视图（before/after）
- HIL 步骤的“人工决策记录”（批准/拒绝/修改内容）

### 4.3 人机协同中的“所见即所得”
当触发 HIL 工单时，用户会看到：
- 当前截图（带标注）
- 计划动作（Action JSON 的可读化展示）
- 风险等级与原因
- 操作按钮：批准/拒绝/修改动作/接管

## 5. 推荐的可视化增强（可选）

- **注意力热力图**：若模型/推理框架可输出注意力/显著性，可渲染热区帮助理解“模型在看哪里”。
- **定位二次确认**：点击前对目标区域进行局部放大预览，让用户更容易判断是否点对。
- **置信度分段阈值**：
  - >=0.9：自动执行
  - 0.8~0.9：执行但提示
  - <0.8：默认进入 HIL
